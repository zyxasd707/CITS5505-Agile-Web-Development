AI Reflection Log

This page documents and critically evaluates how AI tools and large language models (LLMs) were used during the development of this project. It includes example prompts, summaries of AI output, corrections made, and an honest assessment of limitations and ethical considerations.

Overview of AI Usage

Throughout this project, I used Claude (Anthropic) as an AI coding assistant to help with several aspects of development. The AI was used as a collaborative tool rather than a wholesale code generator — I provided direction, reviewed all output, and made significant manual corrections. Below I document three distinct use cases, each illustrating a different way the AI contributed and the limitations I encountered.

palette
Use Case 1: CSS Design System Scaffolding
Generating the initial CSS custom properties and component styles
person My Prompt

“Based on this DESIGN.md specification, create a CSS file with custom properties for the colour palette, spacing scale, border radii, and shadow tokens. Include base reset styles, typography classes using the Manrope font, and component styles for buttons, cards, callouts, and code blocks. Follow the ‘no-line’ rule from the design spec — use background colour shifts instead of borders to separate content.”

smart_toy AI Output Summary

The AI generated a comprehensive CSS file (~400 lines) containing all design tokens as custom properties, a basic CSS reset, typography utility classes, and styled components for buttons, cards, code blocks, and input fields. It correctly translated the colour roles from the design spec into CSS variables and implemented the gradient CTA button as specified.

Corrections & Changes I Made

  • The AI initially used border: 1px solid on several card components, violating the “no-line” rule from the design spec. I replaced these with background colour differentiation using surface container tokens.
  • Some colour contrast ratios did not meet WCAG AA standards. I manually tested using the WebAIM contrast checker and adjusted --on-surface from #2f2f2f to #1a1a1a for better readability.
  • The responsive breakpoints were too aggressive — the AI collapsed to a single-column layout at 900px. I adjusted this to 1024px for tablets and 768px for mobile, which better matched the content density.
code
Use Case 2: Quiz Page JavaScript Logic
AJAX loading, dynamic rendering, scoring, and localStorage
person My Prompt

“Create a quiz.js file that: (1) loads questions from a local JSON file using XMLHttpRequest, (2) shuffles question order with Fisher-Yates, (3) renders questions dynamically as radio-button cards, (4) triggers beforeunload warning after the first answer is selected, (5) validates all questions are answered before submission, (6) calculates score/percentage/pass-fail, (7) on pass, fetches a reward image from the Dog CEO API, (8) saves each attempt to localStorage with date/time.”

smart_toy AI Output Summary

The AI produced a well-structured IIFE (Immediately Invoked Function Expression) with clearly separated functions for each responsibility. It correctly implemented the Fisher-Yates shuffle, XMLHttpRequest-based AJAX loading, dynamic DOM generation, beforeunload event handling, and localStorage persistence. The code included HTML escaping utilities to prevent XSS when rendering question text.

Corrections & Changes I Made

  • The AI used fetch() initially. I changed this to XMLHttpRequest because the assignment specifically required AJAX, and I wanted to demonstrate understanding of the traditional approach. I also added a check for xhr.status === 0 to support the file:// protocol during local development.
  • The beforeunload handler was not being properly removed after submission. The AI attached it with an anonymous function, making removal impossible. I refactored it into a named function (warnBeforeLeave) so it could be cleanly removed via removeEventListener.
  • The reward API response was rendered using innerHTML with the image URL inserted directly, creating a potential injection risk. I added an escapeAttr() utility to sanitise the URL before insertion.
  • The localStorage code lacked error handling for private browsing mode (where setItem throws). I wrapped it in a try-catch to fail silently.

Use Case 3: Tutorial Content Drafting
Generating educational text for the HTML/CSS/JS tutorial page
person My Prompt

“Write educational content for a tutorial page covering three topics: (1) Semantic HTML — explain why semantic elements matter for accessibility, SEO, and maintainability; (2) CSS — cover the Box Model, Specificity, and responsive design with relative units; (3) JavaScript — explain DOM manipulation, event listeners, and pure functions. Include practical code examples for each section.”

smart_toy AI Output Summary

The AI generated well-structured educational text for all three sections, including best-practice callout boxes, bullet-pointed explanations, and inline code examples. The content was technically accurate and appropriately pitched for an introductory web development audience.

Corrections & Changes I Made

  • The AI used overly verbose, textbook-style language. I shortened several paragraphs and rewrote them in a more concise, direct tone to match the site’s editorial voice described in DESIGN.md.
  • One code snippet showed an incorrect <time> element with a malformed datetime attribute (missing closing angle bracket). I caught this during manual review and fixed it.
  • The JavaScript section originally recommended using var exclusively. While this is not incorrect, I revised the explanation to acknowledge modern let/const while keeping var in the demo code for broader compatibility demonstration.

Critical Evaluation

Beyond the specific use cases above, the following reflections capture broader insights about working with AI tools in a coding project.

verified Reliability

AI-generated code was structurally sound roughly 80% of the time, but required manual review for edge cases. The most common failure was subtle logical errors — such as the unreachable removeEventListener bug — that would only surface during real user testing, not code review alone. This reinforces that AI output should never be trusted without thorough testing.

report Limitations

The AI struggled with design-system coherence. When asked to follow the “no-line” rule from DESIGN.md, it still defaulted to border separators in several places, suggesting it cannot reliably internalise and consistently apply nuanced design constraints across an entire codebase. It also produced code that assumed modern browser APIs (like fetch) without considering the assignment’s requirement for traditional AJAX.

balance Ethical Considerations

Using AI in an academic context raises questions of authorship and academic integrity. I ensured AI was used as a productivity tool (similar to Stack Overflow or documentation), not as a substitute for understanding. Every line of AI-generated code was reviewed, understood, and in many cases rewritten. This reflection log itself serves as evidence of that critical engagement. Full AI usage is disclosed in the references section of the CV page.

school What I Learned

The most valuable takeaway is that AI is best used for scaffolding, not finishing. It excels at generating boilerplate, suggesting structural patterns, and producing first drafts of repetitive code. But the “last mile” — ensuring accessibility, security, design consistency, and cross-browser compatibility — still requires human expertise and careful testing. Knowing when to accept, modify, or reject AI suggestions is itself a critical skill.
